# External Integrations

**Analysis Date:** 2026-03-11

## APIs & External Services

**Databricks Workspace APIs:**
- **Purpose:** Core access to Unity Catalog, workspaces, warehouses, jobs, genie spaces
- SDK/Client: `databricks-sdk>=0.60.0` (Python)
- Usage: `src/backend/src/common/workspace_client.py` provides cached `WorkspaceClient` singleton
- Auth: Databricks Personal Access Token (`DATABRICKS_TOKEN` env var) or workspace OAuth
- Key endpoints used:
  - Unity Catalog: tables, schemas, catalogs, volumes, columns
  - Workspace: notebooks, files, jobs, workflows
  - SQL Warehouse: query execution
  - Genie Spaces: space creation (`src/backend/src/common/genie_client.py`)

**Databricks SQL Warehouse:**
- **Purpose:** Ad-hoc query execution for analytics, schema inspection, sample data
- SDK/Client: `databricks-sql-connector>=4.0.0` and `databricks-sqlalchemy>=2.0.0`
- Connection: `DATABRICKS_WAREHOUSE_ID` and `DATABRICKS_HTTP_PATH` env vars
- Usage: `src/backend/src/controller/data_catalog_manager.py`, tools like `src/backend/src/tools/analytics.py`

**Databricks Model Serving (LLM Endpoints):**
- **Purpose:** Claude 3.5 Sonnet serving endpoint for AI-powered features
- Usage: Conversational search, data asset analysis
- Enabled via: `LLM_ENABLED=True` env var
- Endpoint: `LLM_ENDPOINT` (model name only, e.g., `databricks-claude-3-7-sonnet`)
- Base URL: Auto-constructed from `DATABRICKS_HOST` or `LLM_BASE_URL` override
- Authentication: Per-user token from `x-forwarded-access-token` header (in Databricks App) or `DATABRICKS_TOKEN` (local dev)
- Implementation: `src/backend/src/controller/llm_search_manager.py` with tool-calling capabilities
- Capabilities: Search data products, glossary terms, costs, execute analytics queries

**OpenAI API (Deprecated):**
- **Status:** Removed in favor of Databricks Model Serving
- Legacy usage: Comment in `src/backend/src/controller/data_asset_reviews_manager.py` shows `# from openai import OpenAI # Removed OpenAI client`
- Not active in current stack

**Google BigQuery:**
- **Purpose:** External data source connector for asset discovery and metadata retrieval
- SDK/Client: `google-cloud-bigquery>=3.25.0`
- Connector: `src/backend/src/connectors/bigquery.py` (implements `AssetConnector` interface)
- Auth methods (priority order):
  1. UC Connection (extract service account key from Databricks connection)
  2. Inline credentials JSON
  3. Service account key file path
  4. Application Default Credentials (ADC)
- Capabilities:
  - Asset discovery (tables, views, materialized views, external tables, routines, ML models)
  - Metadata retrieval (schema, statistics, ownership)
  - Sample data extraction
  - Validation queries

## Data Storage

**Databases:**

1. **Metadata Database (PostgreSQL):**
   - Type/Provider: PostgreSQL 9.6+
   - Connection vars: `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `DB_SCHEMA`
   - ORM: SQLAlchemy with Alembic migrations
   - Stores: Data products, contracts, glossaries, teams, projects, settings, audit logs, reviews, MDM configs
   - Models: `src/backend/src/db_models/` (SQLAlchemy declarative models)
   - Alternative: Databricks SQL connector with connection string

2. **Databricks SQL Warehouse:**
   - Purpose: Query execution, analytics, operational queries
   - Connection: `DATABRICKS_WAREHOUSE_ID`, `DATABRICKS_HTTP_PATH`
   - Connector: `databricks-sql-connector`

3. **Databricks Unity Catalog (Metadata Lake):**
   - Type: Lakehouse catalog system
   - Stores: Asset metadata, lineage, access logs, audit information
   - Connection: Databricks workspace client with catalog/schema configuration

**File Storage:**

- **Databricks Unity Catalog Volume:**
  - Mounted at: `DATABRICKS_VOLUME` env var (format: `catalog.schema.volume_name`)
  - Uses: App audit logs, Git export storage, file uploads
  - Subdirectories:
    - `audit_logs/` - Audit trail for governance operations
    - `git-export/` - Git repository clone for configuration sync
  - Example: `main.default.app_files` (configured in app.yaml)

- **Frontend Static Assets:**
  - Location: `src/backend/static/` (generated by Vite build)
  - Served: Embedded in Python FastAPI app and deployed with Databricks App
  - Manifest: `manifest.json` for cache busting

**Caching:**

- **In-Memory Caching:**
  - Workspace Client cache: `src/backend/src/common/workspace_client.py` (10-minute TTL)
  - Manager singletons: Stored in FastAPI app state (`app.state`) for request-scoped access
  - Session factory caching: SQLAlchemy session factory cached globally

- **Optional Database Caching:**
  - Concept/taxonomy caching in semantic models (RDFlib in-memory)

## Authentication & Identity

**Auth Provider:** Databricks Native

- **In Databricks App (Production):**
  - Implicit: Uses workspace identity and per-user token from `x-forwarded-access-token` header
  - No explicit login UI (inherited from Databricks workspace)
  - Users identified via `DATABRICKS_USER` header (email)

- **Local Development:**
  - Databricks SDK auto-auth: Tries multiple methods
    1. `DATABRICKS_TOKEN` env var (Personal Access Token)
    2. Databricks CLI profile
    3. OAuth flow (if interactive)

- **Implementation:**
  - `src/backend/src/common/workspace_client.py` - Client instantiation and caching
  - `src/backend/src/common/authorization.py` - Permission checking via group membership
  - User groups fetched from Databricks workspace
  - Roles and RBAC defined in database (managed by `SettingsManager`)

**Role-Based Access Control (RBAC):**

- Roles: Admin, Data Governance Officer, Data Steward, Data Consumer, Data Producer, Security Officer
- Storage: `src/backend/src/db_models/settings.py` (AppRoleDb, RolePermissionDb)
- Mapping: Groups → Roles → Feature access levels (ADMIN, READ_WRITE, READ_ONLY, FULL)
- Default admin groups: `APP_ADMIN_DEFAULT_GROUPS` env var (comma-separated group names)
- Features and access levels defined in `src/backend/src/common/features.py`

## CI/CD & Deployment

**Hosting:**

- Platform: Databricks Apps (serverless container runtime)
- Deployment method: `databricks bundle deploy` (Asset Bundle format in `app.yaml`)
- Manifest: `src/app.yaml` defines entry point, environment, resource bindings
- Database: Auto-injected by Databricks (PGHOST, PGPORT, PGUSER, PGDATABASE)
- SQL Warehouse: Via `valueFrom: "sql-warehouse"` in app.yaml
- Volume: Via `valueFrom: "volume"` in app.yaml
- LLM Endpoint: Via `valueFrom: "serving-endpoint"` in app.yaml

**CI Pipeline:**

- Status: Not configured (no GitHub Actions, GitLab CI, or Jenkins detected)
- Testing: Manual via `pytest` (backend) and `vitest`/`playwright` (frontend)
- Build: Manual via `yarn build` and `hatch build`

**Configuration as Code:**

- YAML deployment: Databricks Asset Bundle format (`src/app.yaml`)
- Variable support: `--var="catalog=app_data" --var="schema=app_ontos"`
- Database migrations: Alembic scripts (auto-run on app startup via `initialize_database`)

## Environment Configuration

**Required Environment Variables:**

- `DATABRICKS_HOST` - Databricks workspace URL (https://your-workspace.cloud.databricks.com)
- `DATABRICKS_TOKEN` - Personal Access Token (optional if using CLI profile or OAuth)
- `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` - Metadata DB credentials
- `DATABRICKS_WAREHOUSE_ID`, `DATABRICKS_HTTP_PATH` - SQL warehouse for queries
- `DATABRICKS_CATALOG`, `DATABRICKS_SCHEMA`, `DATABRICKS_VOLUME` - UC namespace for app data
- `ENV` - Deployment environment (LOCAL, DEV, PROD)

**Optional Configuration:**

- `LLM_ENABLED` - Enable AI features (default: False)
- `LLM_ENDPOINT` - Model serving endpoint name (required if LLM_ENABLED=True)
- `LLM_BASE_URL` - Override serving endpoint base URL
- `LLM_SYSTEM_PROMPT` - Custom system prompt for AI analysis
- `APP_ADMIN_DEFAULT_GROUPS` - Default admin group names (JSON array format)
- `GIT_REPO_URL`, `GIT_BRANCH`, `GIT_USERNAME`, `GIT_PASSWORD` - Git sync configuration
- `SANDBOX_ALLOWED_CATALOGS`, `SANDBOX_ALLOWED_SCHEMAS` - Self-service allowlist (JSON arrays)
- `DEBUG`, `LOG_LEVEL`, `LOG_FILE` - Debug/logging control
- `APP_DEMO_MODE` - Load sample data on startup

**Secrets Location:**

- `.env` file (never committed - in `.gitignore`)
- Databricks App environment variables (injected at runtime)
- Database credentials (in .env or Databricks secrets)
- Git credentials (in .env)
- Google Cloud credentials (service account JSON or Application Default Credentials)

## Background Jobs & Async Work

**Databricks Workflows Integration:**

- Purpose: Heavy lifting (syncing, validation, analysis) offloaded from request cycle
- Management: `JobsManager` (`src/backend/src/controller/jobs_manager.py`)
- Installation: Dynamic via Settings UI, stored as YAML in workspace
- Workflow definitions: `src/backend/src/workflows/` directory
- Trigger: Asynchronous requests return immediately with job ID
- Polling: Frontend polls `/api/jobs/{job_id}` for status
- Notifications: `NotificationsManager` notifies users when jobs complete

**Async Functions:**

- FastAPI async endpoints (`async def`) for SDK calls
- Python `asyncio` for concurrent operations
- No additional async worker queue (Celery/RQ)

## Search & Indexing

**Full-Text Search:**

- Framework: In-application search manager (`SearchManager` in `src/backend/src/controller/search_manager.py`)
- Indexing: Python data structures (lists of SearchIndexItem)
- Sources: Data products, contracts, glossary terms, reviews, assets
- Searchable managers: Decorated with `@searchable_asset` and registered in `SEARCHABLE_ASSET_MANAGERS`
- Type: Simple string matching + filtering (no Elasticsearch/Solr)

**LLM-Powered Conversational Search:**

- Framework: Claude 3.5 Sonnet via Databricks serving endpoint
- Manager: `LLMSearchManager` with tool-calling capabilities
- Tools available: Search products, search glossary, get costs, execute analytics queries
- Session management: In-database conversation history

## Semantic & Knowledge Graph

**RDF/OWL Support:**

- Framework: `rdflib>=7.1.0` for graph processing
- Usage: Ontology loading, concept/property querying, relationship inference
- Ontologies:
  - `src/backend/src/data/taxonomies/ontos-ontology.ttl` (main ontology in Turtle format)
  - GS1 ontology handler
  - Industry ontology manager
- Namespaces: RDF, RDFS, OWL, SKOS, XSD via rdflib
- Storage: In-memory RDFlib graphs loaded at startup

**Semantic Models:**

- Manager: `SemanticModelsManager` (`src/backend/src/controller/semantic_models_manager.py`)
- Operations:
  - Import RDF/Turtle files into database
  - Query ontology concepts and properties
  - Concept search with suggestions
  - Relationship navigation (subclass, subproperty, etc.)
- Database storage: Custom semantic model tables for persistence

## Data Contract Standards

**ODCS (Open Data Contract Standard):**

- Validation: `jsonschema>=4.22.0` with external ODCS JSON schema
- Storage: Text/YAML/JSON formats in database and Databricks Volume
- Export: Generate valid ODCS documents
- Validation: `src/backend/src/schemas/` contains schema definitions

**ODPS (Open Data Product Standard):**

- Specification: BITOL ODPS for data product metadata
- Mapping: Data products mapped to ODPS structure for export
